## Custom Project

### Dataset

The project uses **sales.csv**, containing order transaction records with 15
 fields: order_id, datetime, region_id, currency_code, product_id, unit_price,
 quantity, is_online, customer_id, is_new_customer, device_type,
 payment_method, referral_source, discount_code, and customer_note. The
 original dataset was used without modifications.

### Kafka Messages

The producer sends complete sales records to the Kafka topic. Each message uses
 no explicit key, and message fields remain unchanged from the source dataset.

### Consumer Processing

The consumer receives full sales records from Kafka and processes each message
by creating two output files. It logs message reception and writes records to
both `consumed_sales.csv` (all fields) and `consumed_sales_selected.csv`
(filtered to product_id and region_id only). This demonstrates field selection
and dual output streams.

### Experiments

**Phase 4 change:** Combined two separate LOG.info calls into a single
formatted message: `LOG.info(f"MESSAGE CONSUMED; consumed={consumed_count}")`
to improve log conciseness.

**Phase 5 application:** Added a `select_fields()` function that filters
consumed messages to retain only product_id and region_id, and modified the
consumer loop to write to a separate CSV file containing only these selected
fields while preserving the original full-field output.

### Results

The producer successfully published sales records to Kafka. The consumer
received and processed all records, writing complete records to the primary
output file and filtered records (product and region only) to a secondary file.

### Interpretation

The streaming workflow demonstrated field-level filtering capabilities within
the consumption pipeline. By maintaining parallel outputs—one complete and one
selective—the system showed how Kafka can support multiple downstream consumers
with different data requirements from a single message stream, enabling
flexible data pipelines without requiring separate producers.
